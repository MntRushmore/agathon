[FINDING-01] XSS via renderMarkdown in Journal Page (DOMPurify mitigated)
  File: src/app/journal/[id]/page.tsx
  Lines: 73-148 (renderLatex + renderMarkdown), 1694, 2209
  Severity: LOW
  Description:
    The renderMarkdown() function renders AI-generated chat messages via
    dangerouslySetInnerHTML. However, it DOES apply DOMPurify.sanitize() at
    line 143 with an explicit allowlist of tags and attributes. The renderLatex()
    helper at line 73 passes user-facing content through KaTeX first (which
    generates HTML), then the non-KaTeX portions are HTML-escaped at lines
    107-110 before DOMPurify sanitization. This is a reasonable defense.
  Risk: Low residual risk. The regex-based split at line 102 to separate
    KaTeX output from user text could theoretically be bypassed if KaTeX ever
    produces output not matching the regex pattern. However, DOMPurify at
    line 143 serves as a second line of defense.
  Suggested Fix:
    Consider running DOMPurify on the KaTeX output separately before
    concatenation, rather than relying on the regex split to correctly identify
    all KaTeX-generated spans. Alternatively, use a proper HTML parser instead
    of regex to separate trusted KaTeX output from untrusted user content.

[FINDING-06] Admin Switch-User: Token-Based Re-Authorization
  File: src/app/api/admin/switch-user/route.ts
  Lines: 38-53
  Severity: MEDIUM
  Description:
    When an admin is already impersonating another user and wants to switch
    again, the endpoint accepts an "adminAccessToken" in the request body
    (line 13). This token is verified via supabase.auth.getUser() (line 40),
    which is sound. However, the admin access token is stored client-side
    and sent in the request body, which means it could be intercepted if
    the client stores it insecurely (e.g., localStorage accessible to XSS).
    The endpoint does properly verify the token and check admin role, and
    it logs all impersonation actions to the audit log (line 82).
  Suggested Fix:
    Document the security implications of client-side token storage for the
    switch-user flow. Consider using httpOnly cookies instead of request body
    tokens for the admin token. Add a TTL/expiry check on the admin token.

  Severity: MEDIUM

6. CSRF / CORS ISSUES
================================================================================

[FINDING-15] No Custom CORS Configuration
  Severity: INFO
  Description:
    No Access-Control-Allow-Origin headers are set in any route handler.
    Next.js API routes by default do not add CORS headers, which means
    they are same-origin only. This is the secure default. The CSP in
    middleware.ts further restricts connect-src to specific domains.
  Suggested Fix: No action needed. Same-origin policy is enforced by default.

[FINDING-16] No Explicit CSRF Protection on Mutation Endpoints
  Severity: LOW
  Description:
    POST/PATCH endpoints rely on Supabase auth cookies for authentication.
    Next.js does not automatically include CSRF tokens. However, all mutation
    endpoints require a valid Supabase auth session (verified server-side via
    getUser()), and the session cookie uses SameSite=Lax by default in modern
    browsers, which provides CSRF protection for top-level navigation POST
    requests. The cookie-based auth with SameSite provides adequate protection.
  Suggested Fix: For additional defense in depth, consider adding a custom
    CSRF token header check for state-mutating API routes, especially the
    admin endpoints. Low priority given SameSite cookie protection.

[FINDING-19] No Rate Limiting on Any API Route
  File: All files in src/app/api/
  Severity: HIGH
  Description:
    No rate limiting is implemented on any API route. This includes:
    - AI-powered endpoints that incur third-party API costs:
      * /api/chat (Hack Club AI)
      * /api/solve-math (Hack Club AI)
      * /api/generate-solution (OpenRouter / Hack Club AI)
      * /api/go-deeper (Hack Club AI)
      * /api/check-help-needed (OpenRouter)
      * /api/command-chat (Hack Club AI)
      * /api/ocr (Mistral API)
      * /api/tts (ElevenLabs API)
      * /api/voice/token (OpenAI Realtime API)
      * /api/voice/analyze-workspace (Hack Club AI)
      * /api/journal/generate (Hack Club AI)
      * /api/teacher/generate-feedback (Hack Club AI)
    - Authentication endpoints susceptible to brute force:
      * /api/auth/validate-invite (invite code guessing)
      * /api/auth/redeem-invite
    - Public unauthenticated endpoints:
      * /api/waitlist (email enumeration, spam)
      * /api/help (email spam)
    - Admin endpoints (impersonation):
      * /api/admin/switch-user
      * /api/admin/send-referral-emails (mass email sending)

    An authenticated attacker could run up significant third-party API costs
    by rapidly calling AI endpoints. An unauthenticated attacker could spam
    the waitlist endpoint or attempt to brute-force invite codes.
  Suggested Fix:
    Implement rate limiting using one of these approaches:
    1. Vercel Edge Middleware rate limiting (if deployed on Vercel)
    2. Upstash Redis-based rate limiter (@upstash/ratelimit)
    3. Next.js middleware-level rate limiting
    Recommended tiers:
    - AI endpoints: 20-30 requests/minute per user
    - Auth endpoints: 5-10 requests/minute per IP
    - Public endpoints: 3-5 requests/minute per IP
    - Admin email sending: 1 request/hour per user
